# ADR-004: Целевой движок WebView — только Chromium/CEF; тонкий шов транспорта

- **Статус:** Accepted (пересмотрен 2026-06-27 после углублённой проверки)
- **Дата:** 2026-06-27
- **Связанные требования:** FR-R7, FR-B (транспорт), NFR-3
- **Связанные ADR:** ADR-001 (Runtime), ADR-002 (Bridge), ADR-007 (сборка)

## Контекст
Уточнённые факты (проверка на актуальной документации/релизах ОС Аврора, 2026):

- Актуальный релиз — **Аврора 5.1.6** (апрель 2026, «WebView Chromium 1.8.0»); раскатывается **5.2.0**.
- В разделе «Веб и контент» документации 5.1.6 присутствуют **два** фреймворка:
  **Webview (Chromium)** и **Webview (Gecko)**. Gecko — легаси-путь; для гибридных веб-приложений
  портал разработчика рекомендует **Webview/CEF (Chromium)** («обёртка для веб-приложений на базе CEF»).
- Chromium-WebView Авроры построен на **CEF (Chromium Embedded Framework)**, пакет `ru.auroraos.webview`
  (`pkgconfig(aurorawebview)` / `ru.auroraos.webview-devel`), **не** на QtWebEngine.
- Движок зависит от версии ОС: на 5.1.1 — Gecko, на **5.1.3+ — Chromium**.

Из этого следует ключевой момент про мост (важно для ADR-002):

- **QWebChannel не является нативным мостом** для этого WebView: автоматический транспорт WebChannel
  есть только у QtWebEngine (`QWebEnginePage::setWebChannel()`), которого в CEF-WebView нет. Поднимать
  WebChannel поверх CEF пришлось бы вручную (например, через WebSockets) — это лишний балласт.
- CEF предоставляет **свой штатный асинхронный мост** — `CefMessageRouter`:
  JS-сторона `window.cefQuery({request, onSuccess, onFailure})`,
  C++-сторона `Handler::OnQuery(query_id, request, persistent, callback)` + `callback->Success/Failure`,
  и `CefFrame::ExecuteJavaScript` для native→JS. Это полностью покрывает наши потребности
  (invoke→Promise; `persistent` → стримы/подписки).

## Рассмотренные варианты
1. **Только Chromium/CEF, тонкий шов транспорта (выбран).** Поддерживаем актуальную линейку
   (база 5.1.5/5.1.6+, цель — 5.1.3+ где возможно). Транспорт = один интерфейс + одна реализация
   на `CefMessageRouter` + loopback-двойник для тестов. Gecko **не** реализуем.
2. **Двойная поддержка (Chromium + Gecko).** Совместимость со старыми версиями (5.1.1), но требует
   второй полноценной реализации транспорта (frame scripts) и матрицы тестов = постоянный балласт.
   Отвергнут: целевая аудитория и устройства — на Chromium-линейке; цена поддержки не оправдана.
3. **Жёстко «зашить» CEF без шва.** Минимум кода, но мост становится непроверяемым без устройства и
   хрупким к изменениям CEF/`qtium-driver` (расширяется в 5.2.0). Отвергнут как чуть более рискованный
   при почти нулевой экономии относительно тонкого шва.

## Решение
**Таргетируем только Chromium/CEF-WebView Авроры.** Мост строится на нативном механизме CEF
(`CefMessageRouter` / `window.cefQuery` + `ExecuteJavaScript`). **Gecko-движок не поддерживается**
(минимальная поддерживаемая ОС объявляется на базе Chromium-линейки, ориентир — 5.1.5/5.1.6+).

Сохраняем **тонкий внутренний шов транспорта**: интерфейс `BridgeTransport` + единственная реальная
реализация (CEF) + loopback-реализация для модульных тестов. Это **write-once**, без второй
платформенной реализации. Шов нужен только для тестируемости и изоляции будущих изменений CEF/qtium.

> **Важно для авторов плагинов:** этот шов — исключительно внутренний слой Runtime. Плагины
> взаимодействуют только с контрактом [Native SDK](../architecture/native-sdk.md)
> (`context.resolve/emit/stream`), который лежит **выше** транспорта. Смена/эволюция движка **не
> затрагивает** код плагинов.

## Последствия
- (+) Кодовая база меньше: одна реализация транспорта вместо двух; нет frame-script-слоя Gecko.
- (+) Мост строится на «родном» для CEF `cefQuery` — проще, надёжнее, идеально ложится на Promise/стримы.
- (+) Тонкий шов даёт тестируемость (loopback) и изоляцию изменений CEF/qtium при минимальной цене.
- (+) Авторов плагинов решение не касается вообще.
- (−) Не поддерживаем устройства на Gecko-версиях (напр. 5.1.1). Принимаем осознанно: целевая линейка —
  5.1.3+/5.1.6, это снижает «балласт».
- (−) Привязка к специфике CEF (`cefQuery`) — но изолирована за швом; при необходимости Gecko можно
  добавить позже отдельной реализацией без переписывания верхних уровней моста.

## Заметки/верификация
**Подтверждено официальной документацией** (см. [aurora/webview.md](../aurora/webview.md),
[aurora/verification-status.md](../aurora/verification-status.md), C-1…C-7): движок Chromium/CEF,
доступность Webview (Chromium) с 5.1.3, `pkgconfig(aurorawebview)`, QML-модуль `ru.auroraos.WebView`,
C++ API `Aurora::WebView::*`, наличие `libcef`. `CefMessageRouter`/`window.cefQuery` — штатный механизм CEF.

**V-1 закрыт:** обёртка `ru.auroraos.WebView` даёт собственный async-message API —
`sendAsyncMessage`/`addMessageListener`/`onRecvAsyncMessage`/`runJavaScript` (подтверждено исходниками
демо WebViewAPI, см. [aurora/webview.md](../aurora/webview.md) §5). Транспортный шов строим **поверх него**
(`WebViewTransport` в `packages/bridge-js`); `CefMessageRouter`/`cefQuery` — деталь/fallback.

**Остаётся проверить на SDK/устройстве** (реестр §2): изменения `qtium-driver` в 5.2.0 (**V-4** — закрыт PoC).

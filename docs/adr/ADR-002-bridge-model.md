# ADR-002: Модель Bridge — асинхронный обмен сообщениями

- **Статус:** Accepted
- **Дата:** 2026-06-27
- **Связанные требования:** FR-B1…B9, NFR-1, NFR-2, NFR-7
- **Связанные ADR:** ADR-004 (транспорт/движки), ADR-008 (типы из манифеста)

## Контекст
Нужен надёжный, безопасный и удобный канал JS↔native. Целевой WebView Авроры — **Chromium/CEF**
([ADR-004](ADR-004-webview-engine-abstraction.md)). Нативный мост CEF — **`CefMessageRouter`**:
JS `window.cefQuery({request, onSuccess, onFailure})` ↔ C++ `Handler::OnQuery(... persistent, callback)`
+ `CefFrame::ExecuteJavaScript` для native→JS. Это асинхронная request/response-модель с `query_id`,
идеально ложащаяся на Promise (разовый query) и стримы (`persistent: true`).

> Уточнение: **Qt WebChannel здесь не используется** — он является автоматическим мостом только для
> QtWebEngine (`QWebEnginePage::setWebChannel()`), а не для CEF-WebView Авроры; поднимать его поверх
> CEF (через WebSockets) — лишний балласт. Штатный `cefQuery` покрывает все потребности.

Cordova использует устаревшую callback-модель; Capacitor и Tauri — асинхронный обмен JSON-сообщениями
(Promise/invoke + события), что доказало удобство и безопасность.

## Рассмотренные варианты
1. **Async message passing: Promise (invoke) + события + стримы (выбран).** JSON-сообщения с корреляцией
   по id, ответ как Promise; события «выстрелил-и-забыл»; стримы поверх событий. Как Tauri/Capacitor.
2. **Callback-API в стиле Cordova.** Отвергнут: устаревший DX, «callback hell», слабая типизация.
3. **Синхронный мост (блокирующий).** Отвергнут как основной: блокирует UI-поток, опасен; синхронные
   сообщения допустимы лишь точечно как деталь транспорта.

## Решение
**Асинхронный обмен сообщениями** с двумя примитивами: **Command (invoke→Promise)** и **Event**
(двунаправленный, плюс стримы по подписке). Сообщения JSON-сериализуемы и версионируются
(`bridgeProtocolVersion`). Ошибки — структурированные (код/сообщение/данные) → reject Promise.
Бинарные данные передаются ссылками на ресурсы (через Asset Loader), а не base64 в JSON. Перед
выполнением метода проверяются источник, разрешения, область и схема аргументов.

## Последствия
- (+) Современный DX (async/await), типобезопасность, безопасность (получатель отклоняет «плохие» вызовы).
- (+) Не зависит от конкретного движка: верхние уровни едины, меняется лишь транспорт (ADR-004).
- (+) Единая модель ошибок/событий/стримов.
- (−) Накладные расходы сериализации (минимизируются батчингом и ссылками на ресурсы).
- (−) Требует аккуратной корреляции и управления подписками (таймауты, очистка).

## Заметки/верификация
**Транспорт подтверждён:** интегрируемся на официальном async-API WebView
(`sendAsyncMessage`/`addMessageListener`/`onRecvAsyncMessage`/`runJavaScript`, см.
[aurora/webview.md](../aurora/webview.md) §5; реестр C-13). `invoke→Promise`, события и стримы строятся
поверх него с корреляцией по id и JSON в `data`.

Остаётся: измерить пропускную способность/латентность на реальном устройстве; при необходимости —
бинарный/компактный протокол для горячих путей (FR-B9, отдельный ADR в будущем).

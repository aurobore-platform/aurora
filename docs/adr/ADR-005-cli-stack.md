# ADR-005: Стек CLI и инструментов — Node.js + TypeScript

- **Статус:** Accepted
- **Дата:** 2026-06-27
- **Связанные требования:** FR-C1…C11, FR-S1…S5, NFR-6
- **Связанные ADR:** ADR-008 (кодоген), ADR-007 (сборка)

## Контекст
CLI, SDK и кодогенерация ориентированы на веб-разработчика, живут в npm-экосистеме и тесно связаны с
веб-сборщиками (Vite/Webpack) и TypeScript-типами. Нужно выбрать технологический стек инструментария
(на машине разработчика), отдельный от нативной части (C++/QML на устройстве).

## Рассмотренные варианты
1. **Node.js + TypeScript (выбран).** Один язык с SDK/типами/кодогеном, родная интеграция с npm и
   веб-сборщиками, огромная экосистема, низкий порог входа для целевой аудитории.
2. **Rust (как Tauri CLI).** Быстрее/надёжнее в дистрибуции, но дальше от веб-аудитории, дублирует
   язык относительно TypeScript SDK, выше порог контрибуции.
3. **Python/Go.** Хорошие CLI-языки, но хуже интегрируются с TS-типами/кодогеном и npm-публикацией плагинов.

## Решение
Инструментарий (CLI, build, codegen, dev-server-обвязка, SDK) — на **Node.js + TypeScript**, пакеты
публикуются в npm (`@aurobore/*`). Нативная часть (Runtime, плагины) остаётся на **C++/QML**. CLI
оркеструет нативную сборку, вызывая Aurora SDK (ADR-007), но сам в нативе не реализован.

## Последствия
- (+) Единый язык с SDK/типами; быстрый старт для веб-разработчиков; богатая экосистема.
- (+) Простая публикация плагинов как npm-пакетов; естественная кодогенерация TS.
- (−) Зависимость от Node на машине разработчика (приемлемо для веб-аудитории).
- (−) Производительность CLI ниже, чем у нативного бинаря (некритично для задач оркестрации).

## Заметки/верификация
Минимальная поддерживаемая версия Node и менеджер пакетов (pnpm для workspaces) — зафиксировать в
инфраструктурном чек-листе. Возможна поставка self-contained бинаря CLI в будущем (опция).

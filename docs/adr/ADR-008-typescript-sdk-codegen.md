# ADR-008: Кодогенерация TypeScript SDK из манифестов

- **Статус:** Accepted
- **Дата:** 2026-06-27
- **Связанные требования:** FR-S1…S3, FR-P1, NFR-5
- **Связанные ADR:** ADR-003 (Plugin API), ADR-006 (формат)

## Контекст
Цель — «типизация как продукт» и отсутствие дублирования слоёв. Манифест плагина объявляет методы,
события и типы. Нужно решить, как из этого получаются JS-обёртки и TS-типы, чтобы они никогда не
расходились с нативной реализацией и манифестом.

## Рассмотренные варианты
1. **Генерация JS-обёрток и `.d.ts` из манифеста (выбран).** Манифест — SoT; обёртка и типы —
   артефакты кодогена (`generated/`, «do not edit»). Запуск при `plugin add`/`build`/`generate`.
2. **Ручные обёртки + ручные типы (как Cordova).** Отвергнут: дублирование, расхождения, рутина.
3. **Только рантайм-проксирование без статических типов.** Отвергнут: теряется автодополнение и
   проверка типов на этапе разработки (ключевое преимущество перед Cordova).

## Решение
Из манифеста генерируются: **JS-обёртка** (вызовы `invoke`/подписки) и **TypeScript-типы**
(аргументы/результаты/события), а также декларация разрешений/зависимостей и заготовка доков.
Сгенерированный код помечается как нередактируемый. Любое изменение API начинается с манифеста.

## Последствия
- (+) Слои всегда синхронны (native-регистрация ↔ обёртка ↔ типы ↔ доки).
- (+) Сильная типизация и DX; меньше ошибок интеграции.
- (+) Единый процесс для core- и сторонних плагинов.
- (−) Нужен качественный генератор и валидатор манифеста (тесты на сам кодоген).
- (−) Авторы не правят обёртки напрямую (только манифест + native) — требует обучения/документации.

## Заметки/верификация
Определить целевой формат типов (строгие union/enum из манифеста), стратегию версионирования
сгенерированных пакетов и тесты, проверяющие соответствие «манифест ↔ сгенерированный код ↔ native».

# ADR-006: Формат конфигурации и манифеста

- **Статус:** Accepted
- **Дата:** 2026-06-27
- **Связанные требования:** FR-CF1, FR-P1, NFR-5, NFR-11
- **Связанные ADR:** ADR-003 (Plugin API), ADR-008 (кодоген)

## Контекст
Нужны два декларативных формата: **конфиг проекта** (`aurobore.config`) и **манифест плагина**. Они —
вход для генерации нативных артефактов и кода, должны быть человеко- и машиночитаемы, валидируемы и
версионируемы. Аудитория — веб-разработчики (привычны к JSON/JS/TS-конфигам).

## Рассмотренные варианты
1. **JSON по умолчанию + опциональный TS/JS для конфига; манифест — JSON-подобный со схемой (выбран).**
   JSON прост, повсеместен, легко валидируется JSON Schema; TS/JS-конфиг — для динамики/окружений.
2. **YAML.** Читабелен, но подвержен ошибкам отступов и неоднозначностям типов; меньше предсказуемости.
3. **TOML (как Cargo/Tauri частично).** Хорош для статики, но менее привычен веб-аудитории и хуже для
   вложенных структур манифеста.

## Решение
- **Конфиг проекта:** по умолчанию `aurobore.config.json`; допускается `aurobore.config.(ts|js)` для
  динамических случаев (профили/окружения). Валидируется JSON Schema. Содержит `configVersion`.
- **Манифест плагина:** JSON-подобный декларативный формат со строгой схемой (методы/события/типы/
  разрешения/nativeDeps/compat). Является **единым источником истины** (ADR-003) для кодогена.
- Оба формата **версионируются**; неизвестные поля — ошибка/предупреждение, не молчаливое игнорирование.

## Последствия
- (+) Простая валидация и кодогенерация; предсказуемость и воспроизводимость (NFR-11).
- (+) Привычно веб-аудитории; легко документировать схемой.
- (−) JSON многословен и без комментариев (компенсируется TS-вариантом конфига и документацией схемы).
- (−) Нужна поддержка миграций при изменении формата (`aurobore migrate`).

## Заметки/верификация
Финальные JSON Schema для конфига и манифеста — отдельные спецификации; согласовать поля `nativeDeps`
с реальными требованиями `.spec`/CMake Аврора.

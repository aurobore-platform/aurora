# Агентская архитектура (на базе Composer)

> Соответствует пункту 9 задачи: какие агенты нужны, их обязанности, взаимодействие, skills,
> используемые документы, контекст каждого агента, автоматически делегируемые задачи.
> Это архитектура агентской системы для **разработки самой платформы Aurobore**, без реализации агентов.

## 1. Цель

Использовать **Composer** как основную LLM для ускорения и стандартизации разработки Aurobore:
проектирование, кодогенерация, написание плагинов, документация, тесты, ревью, релизы. Агенты —
специализированные роли с чётким контекстом и набором навыков (skills), которые координируются
оркестратором и опираются на проектную документацию как на источник истины.

> Важно: агенты **не заменяют** ADR-процесс. Архитектурные решения фиксируют люди в ADR; агенты
> действуют в рамках этих решений и предлагают новые через RFC.

## 2. Принципы

1. **Документация — источник истины.** Каждый агент читает релевантные документы `docs/` и обязан им следовать.
2. **Узкая ответственность.** Один агент — одна роль; меньше контекста, выше качество.
3. **Манифест-центричность.** Для плагинов агенты исходят из манифеста (SoT) и не дублируют слои вручную.
4. **Человек в контуре.** Критичные изменения (архитектура, безопасность, релиз) требуют ревью человека.
5. **Прослеживаемость.** Изменения связываются с требованиями (`FR-*`/`NFR-*`), ADR и пунктами roadmap.
6. **Безопасность по умолчанию.** Доступ к командам/сети у агентов ограничен ролью.

## 3. Роли агентов

| Агент | Обязанности | Основной контекст |
|---|---|---|
| **Orchestrator (Координатор)** | Разбивает задачу, выбирает агентов, контролирует выполнение и качество, собирает результат. | `roadmap.md`, `requirements.md`, текущая задача |
| **Architect (Архитектор)** | Поддержка архитектуры, написание/обновление ADR и RFC, оценка влияния изменений. | `architecture/*`, `adr/*`, `rfc/*`, `vision.md` |
| **Runtime Engineer** | Нативный контейнер (C++/QML), движки/транспорт, lifecycle, asset loader. | `architecture/runtime.md`, `bridge.md`, `runtime/` |
| **Bridge Engineer** | Протокол моста, корреляция, события/стримы, сериализация, ошибки, транспорт под движки. | `architecture/bridge.md`, `event-system.md`, ADR-002/004 |
| **Plugin Engineer** | Создание/сопровождение плагинов из манифеста (native + кодоген), conformance. | `plugins/*`, `native-sdk.md`, `plugin-api.md` |
| **SDK/Codegen Engineer** | TypeScript SDK, генераторы JS-обёрток и типов из манифестов. | `architecture/typescript-sdk.md`, ADR-008 |
| **CLI/Build Engineer** | Команды CLI, генерация нативного проекта, упаковка (CMake/RPM), Dev Server. | `architecture/cli.md`, `build-system.md`, `configuration.md`, ADR-007 |
| **Docs Writer** | Документация, справочник API из манифестов, туториалы, changelog. | весь `docs/`, манифесты |
| **Test Engineer** | Модульные/e2e/conformance тесты, фикстуры, матрица совместимости. | `tests/`, `requirements.md`, манифесты |
| **Reviewer (Ревьюер)** | Код-ревью, проверка соответствия ADR/требованиям/стилю, безопасность. | diff, `requirements.md`, `adr/*`, чек-листы |
| **Release Manager** | Версионирование, changelog, публикация пакетов/приложения, релизные чек-листы. | `roadmap.md`, `checklists.md`, CHANGELOG |
| **Research Agent** | Проверка фактов об Аврора/SDK/движках, актуализация допущений, обновление источников. | `vision.md` (источники), внешняя документация |

## 4. Взаимодействие агентов

```
Пользователь / задача
        │
        ▼
  Orchestrator ──┬─► Architect (если нужно решение/ADR/RFC)
                 ├─► профильный инженер (Runtime/Bridge/Plugin/SDK/CLI)
                 ├─► Docs Writer (обновить документацию)
                 ├─► Test Engineer (тесты)
                 └─► Reviewer (ревью результата)  ──► Release Manager (при релизе)
```

- **Модель координации:** Orchestrator делегирует подзадачи специализированным агентам (параллельно,
  где возможно), собирает артефакты, отправляет на ревью.
- **Передача контекста:** каждый агент получает только нужные документы + входной артефакт (манифест/diff/спека).
- **Точки синхронизации:** изменения контракта (протокол/манифест/конфиг) проходят через Architect и Reviewer.
- **Эскалация к человеку:** при архитектурном/безопасностном/релизном решении.

## 5. Skills (навыки)

Skills — переиспользуемые процедуры, которые агенты «читают и выполняют». Предлагаемый набор:

| Skill | Что делает | Кто использует |
|---|---|---|
| `create-plugin` | По манифесту создаёт скелет плагина (native-заготовки контракта + запуск кодогена). | Plugin Engineer |
| `generate-bindings` | Генерирует JS-обёртки и TS-типы из манифеста. | SDK/Codegen |
| `scaffold-template` | Создаёт/обновляет шаблон проекта (`create`). | CLI/Build |
| `write-adr` | Готовит ADR по шаблону `adr/ADR-000-template.md`. | Architect |
| `write-rfc` | Готовит RFC по шаблону. | Architect |
| `update-docs` | Синхронизирует документацию с изменениями (API/конфиг/манифест). | Docs Writer |
| `gen-api-reference` | Собирает справочник API из манифестов. | Docs Writer |
| `add-conformance-test` | Создаёт conformance-тест для метода/события плагина. | Test Engineer |
| `bridge-trace-review` | Анализирует лог bridge-вызовов (DevTools) на ошибки/регрессии. | Bridge Engineer, Reviewer |
| `verify-aurora-deps` | Проверяет имена/версии пакетов Аврора (`ru.auroraos.webview` и т.п.) против SDK. | Research, CLI/Build |
| `release-checklist` | Прогоняет релизный чек-лист и готовит changelog. | Release Manager |
| `requirements-trace` | Связывает изменение с `FR-*`/`NFR-*`/ADR/roadmap. | Orchestrator, Reviewer |

## 6. Какие документы используют агенты

- **Все агенты:** `docs/README.md`, `glossary.md`, `requirements.md` (для трассировки), релевантные чек-листы.
- **Архитектурные:** `architecture/*`, `adr/*`, `rfc/*`, `vision.md`.
- **Плагинные:** `plugins/*`, манифесты плагинов, `native-sdk.md`.
- **Инструментальные:** `cli.md`, `build-system.md`, `configuration.md`, `dev-server.md`, `typescript-sdk.md`.
- **Релизные:** `roadmap.md`, `checklists.md`, `CHANGELOG.md`.

## 7. Контекст каждого агента (что подаётся на вход)

Минимально достаточный контекст (принцип «узкая ответственность»):

- **Роль и границы** (из этого документа).
- **Релевантные документы** (см. таблицу §3).
- **Входной артефакт:** манифест плагина / diff / спецификация задачи / лог ошибок.
- **Связанные требования и ADR** (через `requirements-trace`).
- **Критерии готовности** (соответствующий чек-лист из [checklists.md](../checklists.md)).

## 8. Автоматически делегируемые задачи

Задачи, которые Orchestrator делегирует без ручного вмешательства (с финальным ревью):

- **Кодогенерация** обёрток/типов при изменении манифеста (`generate-bindings`).
- **Синхронизация документации** при изменении публичного API/конфига (`update-docs`, `gen-api-reference`).
- **Создание скелета плагина** из нового манифеста (`create-plugin`).
- **Генерация conformance-тестов** для новых методов/событий (`add-conformance-test`).
- **Трассировка требований** для каждого PR (`requirements-trace`).
- **Проверка зависимостей Аврора** при изменении `nativeDeps` (`verify-aurora-deps`).
- **Подготовка changelog** при релизе (`release-checklist`).
- **Первичное код-ревью** (Reviewer) перед ревью человека.

## 9. Границы и риски

- Агенты не принимают архитектурные решения единолично — только предлагают (RFC) и исполняют (ADR).
- Факты об Аврора/SDK подлежат верификации (`verify-aurora-deps`), т.к. API эволюционирует.
- Доступ к выполнению команд/сети у агентов ограничен ролью; деструктивные действия — с подтверждением.

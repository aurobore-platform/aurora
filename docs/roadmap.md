# Дорожная карта

> Соответствует пункту 11 задачи: этапы и задачи высокого уровня (без расписывания реализации).
> Этапы: Research → Design → MVP → Alpha → Beta → RC → 1.0 → (post-1.0).
> Трассировка к требованиям через идентификаторы `FR-*`/`NFR-*` (см. [requirements.md](requirements.md)).

## Обзор этапов

| Этап | Цель | Главный результат |
|---|---|---|
| 0. Research | Понять платформу и аналоги | Зафиксированные допущения и источники |
| 1. Design | Спроектировать архитектуру | Эта документация + ADR |
| 2. MVP | Доказать жизнеспособность | Работающий минимум: create→dev→build→run + 5 плагинов |
| 3. Alpha | Расширить возможности | HMR, расширенные плагины, шаблоны фреймворков |
| 4. Beta | Стабилизировать | DevTools, conformance-suite, документация, безопасность |
| 5. RC | Подготовить релиз | Заморозка API, аудит, релизные процессы |
| 6. 1.0 | Публичный релиз | Стабильная платформа + экосистема |
| 7. Post-1.0 | Развитие | Доп. плагины, маркетплейс, опц. кросс-платформенность |

---

## Этап 0 — Research (исследование) ✅
- Изучить ОС Аврора: SDK, lifecycle, Qt/QML, WebView (движки по версиям), permissions, упаковку RPM.
- Изучить аналоги: Cordova, Capacitor, Tauri (мост, плагины, процессная модель, безопасность).
- Зафиксировать ключевые ограничения (CEF-WebView; мост через WebView async API, не WebChannel;
  упаковка CMake/RPM).
- **Выход:** раздел источников и допущений в [vision.md](vision.md).

## Этап 1 — Design (проектирование) ✅
- Видение, требования, глоссарий.
- Архитектура всех компонентов (Runtime, Bridge, Plugins, CLI, Build, Dev Server, Events, Config, SDK).
- Структура репозитория, агентская система, документация.
- ADR по ключевым решениям; roadmap и чек-листы.
- Справочник по ОС Аврора ([docs/aurora/](aurora/README.md)) + реестр верификации допущений.
- **Выход:** утверждённая проектная документация (этот каталог). Критерий: команда согласна с ADR-001…009.
- ✅ **Выполнено (2026-06-27):** ADR-001…009 = Accepted; чек-листы [§1/§3](checklists.md) подписаны;
  ключевой техриск — мост на Авроре (V-1) — **закрыт** (API подтверждён офиц. демо, см.
  [aurora/webview.md](aurora/webview.md) §5, [aurora/verification-status.md](aurora/verification-status.md)).

## Этап 2 — MVP 🟢 текущий
Цель — сквозной путь «от веб-приложения до запуска на Аврора». Детальный план — [mvp-plan.md](mvp-plan.md).
- **Runtime:** контейнер + WebView (целевой движок), bootstrap, splash, lifecycle, asset loader, навигация
  (FR-R1…R6).
- **Bridge:** async message passing, Promise, корреляция, базовые события, ошибки, JSON-сериализация,
  транспорт на WebView async API + loopback для тестов (FR-B1…B6).
- **Plugin System:** манифест, статическая регистрация, экспорт API; кодоген обёрток/типов (FR-P1…P4, FR-S1…S3).
- **Плагины MVP:** Device, Storage, FileSystem, Clipboard, Network (FR-P5).
- **CLI:** create, dev (live reload), build, run, plugin add/remove/list, doctor (FR-C1…C6).
- **Config/Build:** `aurobore.config`, генерация нативного проекта (CMake/RPM/.spec/.desktop), сборка пакета
  (FR-CF1…CF3, NFR-11).
- **Docs:** быстрый старт, написание плагина, базовый API; 1 шаблон (vanilla) + 1 пример (FR-D1, FR-D2).
- **NFR:** базовые производительность, безопасность, устойчивость, DX (NFR-1,2,6,7).
- **Критерий выхода:** см. «Критерии приёмки MVP» в [requirements.md](requirements.md).

## Этап 3 — Alpha
- **Bridge+:** стримы с backpressure, эффективная передача бинарных данных/ссылок на ресурсы (FR-B6…B8).
- **Плагины+:** Camera, Geolocation, Notifications (local), Share, Sensors (FR-P6).
- **CLI+:** dev с Hot Reload, plugin create (FR-C7, FR-C8).
- **SDK+:** официальные шаблоны React/Vue/Svelte (FR-S4).
- **Совместимость:** проверка на линейке Chromium/CEF (5.1.5/5.1.6+, 5.2.x); матрица версий ОС
  (FR-R7, NFR-3, ADR-004). Вторая реализация движка (Gecko) **не входит** в область поддержки.
- **Deep links + permissions/scopes** (FR-R8, FR-R9).
- **Критерий выхода:** реальные демо-приложения работают на нескольких версиях Аврора.

## Этап 4 — Beta
- **DevTools:** лог bridge-вызовов, инспекция, профилирование (FR-D3).
- **Тесты:** conformance-suite для плагинов и моста (FR-T1, NFR-8).
- **Безопасность:** аудит модели разрешений/областей/источника моста (NFR-2).
- **Сторонние плагины из npm** + автоинтеграция нативной части (FR-P7).
- **Документация:** полный справочник API из манифестов, туториалы.
- **Стабилизация:** матрица совместимости версий Аврора/движков (NFR-3).
- **Критерий выхода:** внешние команды успешно собирают приложения; API близок к заморозке.

## Этап 5 — RC (Release Candidate)
- Заморозка публичного API и формата конфига/манифеста/протокола.
- Финальный аудит безопасности и производительности.
- `publish` для приложений и плагинов (FR-C9); релизные процессы и версионирование.
- Полный прогон всех [чек-листов](checklists.md).
- **Критерий выхода:** нет известных блокеров; документация и примеры полны.

## Этап 6 — 1.0
- Публичный стабильный релиз пакетов `@aurobore/*` и Runtime.
- Гарантии semver и обратной совместимости в пределах мажора (NFR-5).
- Объявление набора стандартных плагинов как стабильных.
- **Критерий выхода:** платформа пригодна к промышленному использованию.

## Этап 7 — Post-1.0 (развитие)
- Дополнительные плагины: Contacts, Biometric, Secure Storage, Push, Background Tasks (FR-P8).
- Маркетплейс/индекс плагинов и совместимости (FR-C11).
- Бинарный/компактный протокол для горячих путей (FR-B9).
- `migrate` между версиями (FR-C10).
- Опциональная кросс-платформенность (FR-X1) — стратегическое решение через RFC/ADR.

## Сквозные потоки (на всех этапах)
- Документация синхронна с кодом (через кодоген и агентов).
- Тесты растут вместе с функциональностью.
- ADR фиксируют новые решения; спорные идеи проходят через RFC.
- Верификация фактов об Аврора/SDK при каждом значимом изменении нативной части.

> Оценки сроков намеренно не приводятся: приоритет — качество архитектуры и поэтапная проверка
> гипотез на реальной ОС Аврора.

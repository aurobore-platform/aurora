# Аврора SDK: обзор и варианты поставки

> Источники: [`/doc/sdk`](https://developer.auroraos.ru/doc/sdk),
> [`/doc/sdk/app_development`](https://developer.auroraos.ru/doc/sdk/app_development),
> [`/doc/software_development`](https://developer.auroraos.ru/doc/software_development).
> Версия документации: **ОС Аврора 5.2.1 / Аврора SDK 5.2.x**.

## 1. Два набора инструментов

| Набор | Назначение | Раздел документации |
|---|---|---|
| **Аврора SDK** | Прикладная разработка: создание, сборка, запуск, тестирование и отладка **приложений**. Включает документацию по архитектуре и API, примеры и шаблоны. | «Прикладная разработка» |
| **Platform SDK (PSDK)** | Системная разработка: сборка **компонентов ОС** из командной строки; пригоден для CI/CD (в т.ч. GitLab). | «Системная разработка» |

Для Aurobore основной — **Аврора SDK** (сборка приложения), а сценарии CI/упаковки ближе к **PSDK**
(сборка из командной строки, см. [build-and-packaging.md](build-and-packaging.md)).

> Помимо Qt-стека, ОС Аврора официально поддерживает несколько путей доставки веб-контента (сводка —
> [webview.md](webview.md) §8):
>
> - **WebApp Generator** — [официальный OMP-генератор](https://hub.mos.ru/auroraos/tools/WebAppGenerator)
>   WebView-обёрток: JSON-конфиг → готовый нативный проект (Chromium) под **удалённый URL**.
>   Ближайший аналог «мастера в SDK»; не заменяет платформу с мостом и плагинами.
> - **PWA** — установка из браузера, без нативного контейнера разработчика.
> - **Flutter** — адаптированные среда разработки, системные плагины, плагины сообщества, примеры;
>   код Flutter исполняется без изменений.
>
> Для Aurobore Flutter — не целевой путь, но **обязательный референс** при реализации native-плагинов:
> каталог [Flutter Packages на Mos.Hub](https://hub.mos.ru/auroraos/flutter/flutter-community-plugins)
> (в Cursor — `@flutter-community-plugins`). WebApp Generator — референс по packaging/QML-обёртке WebView
> (`.desktop`, cover, permissions). Если нужный плагин есть в каталоге Flutter — смотреть его platform-код
> и логику интеграции с ОС; см. [alpha-plugins-plan.md](https://github.com/aurobore-platform/aurora/blob/main/docs/alpha-plugins-plan.md).

## 2. Варианты поставки Аврора SDK

Документация выделяет два варианта поставки:

| Параметр | **Аврора SDK (mb2)** | **Аврора SDK (BT, Build Tools)** |
|---|---|---|
| Среда сборки | **Aurora OS Build Engine** — окружение, не зависящее от хостовой ОС | **Аврора Build Tools** — инструменты кросс-компиляции |
| Инструмент сборки | **`mb2`** | **`apptool`** |
| Процесс сборки | Кросс-компиляция `gcc` в окружении **таргета**; для каждой архитектуры (x86_64, armv7hl, aarch64) эмулируется окружение сборки | Кросс-компиляция `gcc` в окружении x86_64; для armv7hl/aarch64 поставляется sysroot с библиотеками и исходниками |
| Добавление пакетов | Можно доустанавливать пакеты из официального репозитория (в т.ч. не разрешённые валидатором) | Доустановка из репозитория **недоступна**; разрешённые валидатором библиотеки идут в комплекте, недостающие нужно собирать самому |
| Артефакты сборки | Идентичны | Идентичны |
| Виртуализация среды сборки | Docker | Docker |
| Эмулятор | Aurora OS Emulator (QEMU) | Aurora OS Emulator (QEMU) |
| IDE | Аврора IDE | Аврора IDE |

> Практический вывод для Aurobore: вариант **mb2** удобнее, т.к. позволяет доустанавливать
> зависимости из репозитория (например, пакеты WebView) — это снижает трение при сборке runtime-контейнера.
> Точные имена этих пакетов — см. [webview.md](webview.md) `(verify)`.

## 3. Целевые архитектуры

- `x86_64` — эмулятор;
- `armv7hl` — 32-битные ARM-устройства;
- `aarch64` — 64-битные ARM-устройства.

Таргет именуется по схеме `AuroraOS-<номер_релиза>-base-<arch>`
(пример: `AuroraOS-4.0.2.89-base-armv7hl`). Это имя передаётся в `mb2 --target ...` и `sb2 -t ...`.

## 4. Аврора IDE и эмулятор

- **Аврора IDE** — среда на базе Qt Creator для C/C++/QML с компонентами Silica: редактор,
  интеграция VCS, управление проектами и сборками.
- **Aurora OS Emulator** — виртуальная машина (QEMU), исполняющая приложения в окружении ОС Аврора
  аналогично реальному устройству (МУ — «мобильное устройство»).

> ⚠️ С Аврора SDK **5.2.0+** не рекомендуется запускать SDK внутри виртуальной машины: эмулятор
> внутри ВМ работает медленнее и может вести себя неопределённо. Запускать SDK следует на хосте.

## 5. Что это значит для Aurobore

- CLI Aurobore (`build`/`run`) должен оперировать понятием **таргета** (`AuroraOS-<rel>-base-<arch>`)
  и уметь выбирать архитектуру — см. [architecture/cli.md](../architecture/cli.md).
- Сборка финального приложения делегируется штатному `mb2`/`apptool` — Aurobore лишь генерирует
  корректный нативный проект (CMake/RPM), см. [architecture/build-system.md](../architecture/build-system.md).
- Для Dev Server и быстрой итерации целевая среда запуска — **эмулятор** (x86_64), см.
  [architecture/dev-server.md](../architecture/dev-server.md).

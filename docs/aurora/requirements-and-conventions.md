# Требования и соглашения ОС Аврора

> Источник: [`/doc/software_development/guidelines`](https://developer.auroraos.ru/doc/software_development/guidelines/webview)
> (раздел «Рекомендации и требования»). Версия документации: **ОС Аврора 5.2.1**.

Раздел официальной документации задаёт обязательные требования и рекомендации к прикладному ПО для
ОС Аврора — ради единообразия, надёжности, безопасности и сопровождаемости кода, а также корректной
сборки, установки и работы на устройствах.

## 1. Состав требований (по документации)

| Тема | Содержание |
|---|---|
| **Рекомендации к проектам ПО** | Общие подходы: требования к интерфейсу, функционированию, сборке, упаковке и безопасности; перечень ожидаемых артефактов разработки. |
| **Профили безопасности** | Автоматический выбор профиля безопасности при **валидации пакета** в зависимости от **типа сертификата подписи**. |
| **Чек-лист готовности функциональности** | Структурированный перечень критериев для проверки новой функциональности, исправления дефектов или завершения разработки. |
| **Установочные пакеты** | Требования к RPM для корректной установки/работы, **обязательное подписание** сертификатами. Включает оформление **`.spec`** и **`.desktop`**. |
| **Оформление исходного кода** | Соглашения по C++ (с Qt) и QML: читаемость, тестируемость, сопровождаемость. |
| **Конфигурация проекта с CMake** | Автоматизация сборки через `CMakeLists.txt`. |
| **Конфигурация проекта с qmake** | Генерация Makefile из `.pro`-файлов. |

## 2. Профили безопасности и подпись

- Профиль безопасности выбирается **автоматически** при валидации пакета и зависит от **типа
  сертификата**, которым подписан RPM.
- Подпись пакета **обязательна** (см. процесс в [build-and-packaging.md](build-and-packaging.md) §3).
- Вывод для Aurobore: канал публикации/тип сертификата влияет на доступные приложению возможности —
  это нужно учитывать в документации по релизу и в `doctor`-проверках.

## 3. Оформление `.spec` и `.desktop`

- Для обоих артефактов есть **отдельные требования к оформлению**.
- `.desktop` несёт не только метаданные запуска, но и **декларацию разрешений** (см.
  [sandbox-and-permissions.md](sandbox-and-permissions.md)).
- Aurobore генерирует эти файлы из конфига — генерация обязана соответствовать требованиям ОС
  ([architecture/configuration.md](../architecture/configuration.md), [adr/ADR-007](../adr/ADR-007-packaging-build.md)).

## 4. Стиль кода C++/QML

Эти соглашения релевантны прежде всего для:

- **нативного контейнера/транспорта** (`runtime/`),
- **авторов нативных плагинов** ([architecture/native-sdk.md](../architecture/native-sdk.md)).

Сгенерированный Aurobore нативный код должен следовать этим соглашениям, чтобы проходить ревью/валидацию.

## 5. Проекция на наши чек-листы

Официальный «чек-лист готовности функциональности» — внешний эталон для нашего
[checklists.md](../checklists.md): при сборке релизных критериев Aurobore стоит сверяться с критериями ОС
Аврора (упаковка, подпись, разрешения, стиль кода).

## 6. Что верифицировать `(verify)`

- Конкретные правила оформления `.spec`/`.desktop` для версии 5.2.x.
- Перечень профилей безопасности и сопоставление «тип сертификата → профиль → доступные возможности».
- Актуальные правила валидатора пакетов (какие библиотеки/зависимости разрешены).
- Действующий гайд по стилю C++/QML (версия документа ОМП).

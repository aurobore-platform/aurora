## I. Скорость inner-loop (DX)

### I.1 — Браузерный mock-режим (`aurobore dev --web`) ✅

> **Статус:** готово. `FR-C12` в [requirements.md](requirements.md); см. [web-mock-mode.md](dev/web-mock-mode.md),
> `tools/aurora/verify-web-dev.mjs`.

**Мотивация.** Сейчас цикл разработки завязан на эмулятор/устройство Аврора. Большая часть UI-работы
веб-разработчика не требует нативного слоя. Нужен запуск приложения в обычном desktop-браузере с
**мок-реализациями** плагинов, чтобы итерации по вёрстке/логике шли без SDK.

**Область.** `packages/cli` (флаг `dev --web`), `packages/bridge-js` (уже есть loopback-двойник транспорта —
переиспользовать), `packages/core` (регистрация мок-хендлеров), опционально `templates/*` (dev-манифест
моков). Мок insets (`--aurobore-safe-area-*`) — CSS-инъекция, как в A2.

**Предлагаемый ID.** `FR-C12`.

**Зависит от.** Loopback-транспорт (A1/`runtime/transport/loopback`), контракт моста (M2).

**Критерий готовности.** `aurobore dev --web` открывает приложение в браузере; вызовы плагинов возвращают
детерминированные mock-ответы (напр. Camera → фикстура-изображение, Geolocation → заданные координаты);
переключение на реальный контейнер не требует изменений кода приложения.

### I.2 — Dev-client «Aurobore Go» (превью по сети) ⬜

> **Статус:** не начато. Нужны RFC/ADR (см. [открытые вопросы](#открытые-вопросы-нужны-rfcadr)).

**Мотивация.** Установка на устройство требует пересборки RPM и подписи на каждой итерации. Generic-контейнер,
который один раз ставится на устройство и грузит **любой** web-бандл с dev-сервера (по QR/коду), убирает
пересборку из inner-loop (аналог Expo Go).

**Область.** `runtime/container` (сборочный вариант dev-client: загрузка entry по сетевому URL вместо
локального asset-origin, с явной пометкой «dev/insecure»), `packages/cli` (генерация QR/кода подключения,
интеграция с dev-сервером и туннелем до эмулятора).

**Предлагаемый ID.** `FR-C13` (CLI-сторона), `FR-R12` (runtime dev-client).

**Зависит от.** Dev Server + диагностика доступности хоста (A4), asset-loader/навигация (M1).

**Критерий готовности.** Установленный dev-client открывает проект по коду/QR; изменения веба видны на
устройстве без пересборки RPM; режим явно отделён от продакшн-контейнера (не грузит удалённый код в релизе).

## II. Доставка обновлений

### II.1 — OTA / Live Updates веб-бандла ✅

> **Статус:** MVP готов. `FR-R13`/`FR-C14` в requirements (COULD); [ADR-012](adr/ADR-012-ota-live-updates.md),
> [ota-updates.md](dev/ota-updates.md), `examples/ota-demo`, `UpdateManager` в `runtime/container`.
> **Осталось (не блокирует MVP):** sign-off QCA Ed25519 на SDK 5.2.x; delta-обновления — вне Alpha.

**Мотивация.** Приложение по сути — веб-бандл. Обновления веб-части можно доставлять без переиздания RPM в
стор: каналы (`stable`/`beta`), атомарное применение, откат, **подпись бандла**. Для WebView-платформ это
киллер-фича (ср. Capacitor Live Updates, Expo Updates); в текущем roadmap её нет.

**Область.** `runtime/container` (проверка/скачивание/атомарная подмена активного бандла, откат, проверка
подписи), `packages/build` (формирование подписанного OTA-бандла с манифестом версии), `packages/cli`
(`aurobore update publish|list|rollback`, управление каналами), `packages/core` (события `update:available` /
`update:ready`, API применения). Совместимо с моделью `minOs`/версий из A6.

**Предлагаемый ID.** `FR-R13` (runtime), `FR-C14` (CLI/каналы).

**Зависит от.** Детерминированная сборка веб-части (M4/`build-system`), безопасная схема ресурсов (M1),
модель версий/совместимости (NFR-5). Требует **ADR** (безопасность, доверие к источнику, откат).

**Критерий готовности.** CLI публикует подписанный бандл в канал; контейнер обнаруживает и применяет
обновление с проверкой подписи; при сбое — откат к предыдущему бандлу; событие доходит до JS. Обновляется
только веб-часть; смена нативного слоя/permissions по-прежнему через RPM.

## III. Совместимость с веб-стандартами

### III.1 — Полифилы W3C API поверх нативных плагинов 🟡

> **Статус:** частично. `FR-S6` в requirements; `@aurobore/polyfills` + `examples/w3c-demo`.
> **Готово:** geolocation, share, notification, clipboard (дефолтный набор).
> **Осталось:** device-спайк `V-webview-getusermedia` — engine-first getUserMedia; Camera-fallback
> (`mediaDevices` opt-in) заблокирован до резолюции ([ADR-011](adr/ADR-011-camera-frame-channel.md)).

**Мотивация.** Прямое попадание в цель **G1** и аудиторию №2 («переносящие SPA/PWA», [vision.md](vision.md)):
существующее веб-приложение должно работать **без изменения кода**. Стандартные браузерные API маппятся на
наши плагины:

| Web-стандарт | Плагин Aurobore |
|---|---|
| `navigator.geolocation` | Geolocation |
| `navigator.share` (Web Share) | Share |
| `Notification` | Notifications (local) |
| `navigator.mediaDevices.getUserMedia` / `<input capture>` | Camera |
| `navigator.clipboard` | Clipboard |

**Область.** `packages/core` или отдельный `packages/polyfills` (`@aurobore/polyfills`), подключаемый
опционально при bootstrap. Не нарушает не-цель **N2** (не UI-фреймворк): это адаптеры API, а не компоненты.

**Предлагаемый ID.** `FR-S6`.

**Зависит от.** Плагины A3 (Camera/Geo/Notifications/Share) и MVP Clipboard; модель ошибок SDK (FR-S3).

**Критерий готовности.** Демо-SPA, использующее только стандартные `navigator.*`/`Notification`, работает в
контейнере без правок; недоступные возможности деградируют предсказуемо (feature-detection, а не throw).

### III.2 — Импорт `webmanifest` → `aurobore.config` ⬜

> **Статус:** не начато.

**Мотивация.** У PWA уже есть `manifest.webmanifest` (имя, иконки, ориентация, `theme_color`, обработчики
протоколов). Автогенерация/слияние с `aurobore.config` снижает порог переноса до одной команды.

**Область.** `packages/build` (парсер + маппинг полей → конфиг/иконки/`deepLinks`), `packages/cli`
(`aurobore config import <manifest>` или шаг в `create`).

**Предлагаемый ID.** `FR-CF4`.

**Зависит от.** Формат `aurobore.config` и генерация иконок (M4/`configuration`).

**Критерий готовности.** Из валидного `webmanifest` генерируется корректный `aurobore.config` (имя,
ориентация, иконки, схемы протоколов → `deepLinks`); конфликтующие поля разрешаются с понятным выводом.

## IV. Качество и наблюдаемость

### IV.1 — Плагин Telemetry / Crash Reporting ⬜

> **Статус:** не начато. Каталог `plugins/telemetry` отсутствует.

**Мотивация.** Для WebView-приложений важно видеть и JS-ошибки, и падения нативного контейнера. Плагин
собирает необработанные ошибки/крэши и отдаёт их приложению (или во внешний sink через конфиг). Дополняет
NFR-12 (наблюдаемость) на уровне приложения, а не только dev-логов.

**Область.** `plugins/telemetry` (манифест + native: перехват `unhandledrejection`/`window.onerror` на JS,
хук на нативные фатальные сигналы контейнера), `runtime/native-sdk` (контракт репорта). Символикация
нативных крэшей — отдельная под-задача.

**Предлагаемый ID.** `FR-P9`.

**Зависит от.** Plugin System (M3), стримы/события (A1). Приоритет — Post-Beta (COULD).

**Критерий готовности.** JS-ошибка и симулированный нативный крэш доходят до обработчика приложения с
контекстом (версия, платформа); отправка во внешний sink опциональна и конфигурируема; выключено по умолчанию
(приватность).

### IV.2 — Bundle analyzer / performance budget ⬜

> **Статус:** не начато. Флаг `build --analyze` отсутствует.

**Мотивация.** Железо Аврора слабее флагманов. Проверка размера/структуры веб-бандла и бюджета
(размер, число ресурсов, крупные зависимости) прямо в `build`/CI предотвращает регрессии производительности.

**Область.** `packages/build` (анализ артефактов сборки, отчёт), `packages/cli` (флаг `build --analyze`,
пороги из `aurobore.config`, ненулевой код возврата при превышении бюджета в CI).

**Предлагаемый ID.** `FR-C15`.

**Зависит от.** Build System (M4).

**Критерий готовности.** `aurobore build --analyze` печатает отчёт по размеру бандла и крупнейшим модулям;
при превышении заданного в конфиге бюджета сборка падает (для CI).

### IV.3 — E2E / device test harness ✅

> **Статус:** готово (minimal). `FR-T2` в [requirements.md](requirements.md); `pnpm e2e:verify` →
> [`tools/aurora/verify-e2e.mjs`](../tools/aurora/verify-e2e.mjs); сценарий [`tests/e2e/README.md`](../tests/e2e/README.md).
> Env-gated assert в [`WebAppPage.qml`](../runtime/container/qml/pages/WebAppPage.qml) (`AUROBORE_E2E=1`).

**Мотивация.** `conformance-suite` (FR-T1, Beta) проверяет плагины; отдельно нужен способ гонять **UI/e2e**
сценарии приложения на эмуляторе (драйвер поверх `runJavaScript` + проверка ответов моста).

**Область.** `tests/e2e` (сценарии create→build→run→проверка), `tools/aurora` (драйвер: отправка команд в
WebView, снятие результата, ассерты; переиспользовать `container:*`/`native-debug.mjs`).

**Предлагаемый ID.** `FR-T2`.

**Зависит от.** Dev-toolkit (`tools/aurora`, M0.5), контейнер (M1). Связано с NFR-8 (тестируемость).

**Критерий готовности.** Один прогон гоняет сквозной сценарий на эмуляторе и ассертит поведение моста/UI;
воспроизводимо, без ручной оркестрации команд в терминале (см. правило demo-verification).

## V. Схемы и цепочка поставок

### V.1 — JSON Schema для `aurobore.config` и `plugin.manifest` ✅

> **Статус:** готово. `FR-D5` в [requirements.md](requirements.md); схемы в
> [`packages/build/schema/`](../packages/build/schema/); `$schema` в `templates/*`; см.
> [configuration.md](architecture/configuration.md) §7.

**Мотивация.** Публикуемая JSON Schema даёт автодополнение и валидацию в **любом** редакторе через
`$schema` — **без** отдельного IDE-расширения. Дёшево, заметно улучшает DX и снижает ошибки конфигурации.

**Область.** `packages/build` (генерация/публикация схемы из типов SoT + валидатор), `docs` (ссылка на схему,
инструкция по `$schema`). Опционально — публикация схемы на стабильном URL для `$schema`.

**Предлагаемый ID.** `FR-D5`.

**Зависит от.** Формат конфига/манифеста (M3/M4). Валидация конфига уже есть (`packages/cli` `config`).

**Критерий готовности.** `aurobore.config`/`plugin.manifest` с `$schema` дают автодополнение и подсветку
ошибок в редакторе без плагинов; схема публикуется вместе с релизом.

### V.2 — Цепочка поставок: SBOM, подпись плагинов, permission-diff ⬜

> **Статус:** не начато. Нужен ADR (см. [открытые вопросы](#открытые-вопросы-нужны-rfcadr)).

**Мотивация.** С приходом сторонних плагинов из npm (FR-P7, Beta) нужна прозрачность и контроль: что именно
получает доступ к нативу. Современный инструмент показывает изменения разрешений и проверяет целостность.

**Область.** `packages/cli` (`plugin add` показывает **diff** `permissions`/`scopes`, которые добавятся в
`.desktop`; проверка подписи/интегрити плагина), `packages/build` (генерация SBOM для RPM).

**Предлагаемый ID.** `NFR-13` (расширяет NFR-2 «безопасность»).

**Зависит от.** Модель разрешений/scopes (A2), сторонние плагины (FR-P7). Требует **ADR** (модель доверия к
плагинам).

**Критерий готовности.** `plugin add <name>` показывает список новых разрешений до применения; сборка
опционально формирует SBOM; неподписанный/несовпадающий плагин отклоняется или требует явного подтверждения.

## Вне области (не в этом репозитории)

Следующие идеи ценны, но реализуются **отдельными продуктами/инфраструктурой**, а не в этом монорепо, и
поэтому в план не включены:

| Идея | Почему вне репо |
|---|---|
| MCP-сервер / AI-tooling для Aurobore | Отдельный продукт поверх CLI, не код платформы |
| Расширение VS Code / Cursor (launch/debug, автодополнение) | Отдельный репозиторий расширения; автодополнение конфига уже покрывается JSON Schema (V.1) через `$schema` |
| Hosted cloud build & signing (SaaS, аналог EAS) | Внешний сервис/инфраструктура; в репо может жить лишь воспроизводимый CI/Docker-образ сборки, но не сам сервис |
| Плагин-маркетплейс / индекс (FR-C11) | Внешний сервис; уже помечен как Post-1.0 в roadmap |

> Примечание: воспроизводимая **контейнеризованная сборка RPM в CI** (Dockerfile + workflow) — это уже
> артефакт репозитория (`tools/`/CI) и может быть добавлена отдельным пунктом; здесь она не расписана, так
> как относится к инфраструктуре сборки, а не к возможностям инструмента для конечного разработчика.

  - **Резолюция getUserMedia — engine-first, plugin-fallback** ([RFC-001](rfc/RFC-001-w3c-polyfills.md),
    [ADR-011](adr/ADR-011-camera-frame-channel.md)). `getUserMedia` — API движка WebView, поэтому `mediaDevices`
    **исключён из дефолтного набора** полифилов (`web.polyfills: true` = 4 лёгких адаптера) и включается только
    явным opt-in. Camera-fallback (`watchPreview` + бинарные кадры) **заблокирован на device-спайк**
    `V-webview-getusermedia` ([verification-status](aurora/verification-status.md) §2): на устройстве с камерой
    проверить нативный getUserMedia и возможность выдать media-permission из натива.
- **Supply chain (V.2):** модель доверия к сторонним плагинам, формат/место хранения подписей. → ADR.
- **Aurobore Go (I.2):** безопасность загрузки удалённого кода в dev-режиме, чёткое отделение от релиза.

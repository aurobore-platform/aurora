# Видение проекта

## 1. Краткое описание

**Aurobore** — открытая платформа для создания мобильных приложений под **ОС Аврора**
с использованием стандартных веб-технологий (HTML / CSS / JavaScript / TypeScript) и доступом
к нативным возможностям системы через типизированный асинхронный мост и расширяемую систему плагинов.

По духу это аналог **Capacitor / Cordova / Tauri**, но спроектированный «с нуля» под особенности
Аврора (Qt/QML + C++, WebView на базе Chromium/CEF, упаковка в RPM, модель разрешений Аврора).

Цель — не «обёртка над WebView», а полноценная среда разработки: единый runtime, удобный мост
между вебом и нативом, расширяемая экосистема плагинов, CLI, dev-сервер и официальный SDK.
Aurobore должен занять для Аврора ту же нишу, что Capacitor занимает для Android/iOS.

## 2. Проблема

На сегодняшний день, чтобы выпустить приложение для ОС Аврора, разработчик обязан работать
с нативным стеком: C++, QML, Qt, Aurora SDK, CMake/qmake, RPM, `.spec`- и `.desktop`-файлы.
Порог входа высок, а огромное количество существующих веб-приложений и команд с веб-компетенциями
не могут переиспользовать свой код и навыки.

Прямая адаптация веб-приложения через нативный WebView возможна (фреймворк поддерживается в Аврора),
но требует ручного написания нативного контейнера, моста, обработки жизненного цикла, разрешений и
сборки под каждый проект заново. Это и есть та рутина, которую устраняет Aurobore.

## 3. Цели (Goals)

- **G1.** Дать веб-разработчику возможность создать, запустить и собрать приложение под Аврора
  без написания C++/QML вручную.
- **G2.** Предоставить **типизированный** (TypeScript-first) асинхронный API к нативным возможностям.
- **G3.** Сделать систему плагинов первоклассной сущностью: плагины ставятся, обнаруживаются и
  регистрируются автоматически, без ручного редактирования нативных конфигов.
- **G4.** Дать удобный инструментарий разработчика: CLI, dev-сервер, live/hot reload, диагностика (`doctor`).
- **G5.** Заложить версионирование и совместимость, чтобы Runtime, Bridge и плагины могли
  эволюционировать независимо.
- **G6.** Сделать проект устойчивым к различиям версий Аврора в пределах Chromium/CEF-линейки
  (5.1.5/5.1.6+, 5.2.x), изолируя специфику движка за тонким швом транспорта.
- **G7.** Построить здоровую open-source экосистему: документация, шаблоны, примеры, тесты совместимости.

## 4. Не-цели (Non-Goals)

- **N1.** Не делаем кросс-платформенный runtime «для всего» (Android/iOS/Web) на первом этапе.
  Фокус — Аврора. Кросс-платформенность — возможное будущее, но не сейчас.
- **N2.** Не делаем UI-фреймворк/набор компонентов. UI — ответственность веб-приложения.
- **N3.** Не пытаемся скопировать архитектуру Cordova 1:1 (устаревшая callback-модель, ручная регистрация).
- **N4.** Не заменяем Aurora SDK — мы его используем (генерируем нативный проект и вызываем штатную сборку).
- **N5.** Не поддерживаем синхронный «тяжёлый» мост по умолчанию (мост асинхронный, см. [ADR-002](adr/ADR-002-bridge-model.md)).

## 5. Целевая аудитория

1. **Веб-разработчики**, желающие выпускать приложения под Аврора без глубокого нативного стека.
2. **Команды**, переносящие существующие SPA/PWA на Аврора.
3. **Авторы плагинов** — нативные разработчики, расширяющие платформу новыми возможностями.
4. **Сообщество и интеграторы** Аврора, которым нужен стандартный быстрый путь поставки веб-приложений.

## 6. Ключевые архитектурные принципы

1. **Async-first.** Только Promise / async-await и подписки. Никаких голых callback-интерфейсов
   как основного API (callback допустим лишь как низкоуровневая деталь транспорта).
2. **Типизация как продукт.** TypeScript SDK с автодополнением и строгими типами — первоклассная фича,
   а не дополнение. Типы и JS-обёртки **генерируются** из манифеста плагина (single source of truth).
3. **Плагины — это контракты.** Плагин описывается манифестом (методы, события, типы, разрешения).
   Из манифеста выводится всё: native-регистрация, JS-обёртка, типы, документация.
4. **Целевой движок — Chromium/CEF.** Мост строится на WebView async API (`sendAsyncMessage`/…,
   `ru.auroraos.WebView`). Gecko (легаси, версии до 5.1.3) не поддерживается; транспорт скрыт за тонким
   внутренним швом ради тестируемости и изоляции изменений CEF. См. [ADR-004](adr/ADR-004-webview-engine-abstraction.md).
5. **Безопасность по умолчанию.** Доступ к нативу ограничен разрешениями и областями (scopes),
   мост проверяет источник и допустимость вызовов (вдохновлено permissions/capabilities Tauri).
6. **Конвенции вместо конфигурации.** Разумные значения по умолчанию; конфиг нужен только для отклонений.
7. **Воспроизводимость и автоматизация.** Сборка, генерация нативного проекта и упаковка — детерминированы
   и автоматизированы CLI; ручное редактирование сгенерированных артефактов не требуется.
8. **Эволюционируемость.** Версионируются: формат конфига, протокол моста, API плагинов и Runtime.

## 7. Высокоуровневая архитектура (вид с высоты)

```
┌──────────────────────────────────────────────────────────────┐
│                     Web-приложение (HTML/CSS/JS/TS)            │
│                  использует @aurobore/* (TypeScript SDK)       │
└───────────────────────────┬──────────────────────────────────┘
                            │  типизированные вызовы / подписки
┌───────────────────────────▼──────────────────────────────────┐
│                         BRIDGE (JS-сторона)                    │
│   сериализация, корреляция запрос↔ответ, Promise, события      │
└───────────────────────────┬──────────────────────────────────┘
       транспорт на WebView async API (sendAsyncMessage / runJavaScript)
      (Chromium/CEF) за тонким внутренним швом — ADR-004
┌───────────────────────────▼──────────────────────────────────┐
│                  RUNTIME (нативный, C++/QML)                   │
│  WebView (Chromium/CEF) • Lifecycle • Navigation • Splash •    │
│  Asset Loader • Permissions • Deep Links • Bridge (native)     │
└───────────────────────────┬──────────────────────────────────┘
                            │  маршрутизация вызовов
┌───────────────────────────▼──────────────────────────────────┐
│            PLUGIN MANAGER / PLUGIN LOADER                      │
│        реестр плагинов, диспетчеризация методов/событий        │
└───────────────────────────┬──────────────────────────────────┘
                            │
┌───────────────────────────▼──────────────────────────────────┐
│      НАТИВНЫЕ ПЛАГИНЫ (Camera, FS, Geo, Notifications, …)      │
└───────────────────────────┬──────────────────────────────────┘
                            │
┌───────────────────────────▼──────────────────────────────────┐
│                  Qt / Aurora Public APIs / ОС Аврора          │
└──────────────────────────────────────────────────────────────┘
```

Вокруг этого ядра работают **инструменты разработчика**: CLI, Build System, Dev Server (live/hot reload),
а также **агентская система** на базе Composer для автоматизации разработки самой платформы.

## 8. Что мы делаем лучше, чем Cordova

| Cordova (legacy) | Aurobore |
|---|---|
| Callback-API | Async-first: Promise / async-await / подписки |
| Ручная регистрация плагинов, правка конфигов | Автообнаружение и автогенерация регистрации |
| Слабая типизация | TypeScript-first, типы из манифеста |
| Дублирование JS-обёрток вручную | Кодогенерация JS-обёрток и типов из манифеста |
| Жёсткая привязка к движку | Абстракция движка WebView (важно для Аврора) |
| Минимум средств отладки | DevTools: лог моста, инспекция вызовов, профилирование |
| Слабая модель безопасности | Разрешения + области (scopes) на уровне моста |

## 9. Результат этапа проектирования

Готовая, согласованная проектная документация (этот каталог `docs/`), которой достаточно,
чтобы начать поэтапную реализацию по [дорожной карте](roadmap.md) без переоткрытия ключевых решений.

## 10. Источники исследования

Документация и материалы, использованные при проектировании:

- Официальная документация ОС Аврора — <https://developer.auroraos.ru/> (разделы «Разработка ПО»,
  «Подключение зависимостей», «Начало работы с QML», публичные API).
- Пример WebView-приложения Аврора (CEF) — Mos.Hub `auroraos/demos/WebViewBrowser`.
- Релизы ОС Аврора: **5.1.6** (03.04.2026, «WebView Chromium 1.8.0»), 5.1.5; раскатка **5.2.0**
  (developer.auroraos.ru/release_notes). Раздел «Веб и контент» 5.1.6: фреймворки
  **Webview (Chromium)** (CEF, `ru.auroraos.webview`) и **Webview (Gecko)** (легаси).
- **CEF (Chromium Embedded Framework)** как основа Chromium-WebView Авроры; под wrapper-API —
  `sendAsyncMessage`/`runJavaScript` ([aurora/webview.md](aurora/webview.md) §5); `CefMessageRouter`/`cefQuery` — уровень ниже.
- Различия движков по версиям (Gecko на 5.1.1, Chromium с 5.1.3+) и переход qmake → CMake,
  точные RPM-имена (`ru.auroraos.webview`, `pkgconfig(aurorawebview)`) — блог D. Kazakov, 2025.
- Уточнение по WebChannel: автоматический транспорт `QWebChannel` существует только у **QtWebEngine**
  (`QWebEnginePage::setWebChannel`), а **не** у CEF-WebView Авроры — поэтому мост строится на
  WebView async API, а не на WebChannel.
- (Легаси/для контекста) Sailfish/Gecko WebView: `loadFrameScript`, `sendAsyncMessage` /
  `recvAsyncMessage` — не используется в выбранной архитектуре, оставлено как справка.
- Документация Capacitor — модель `window.Capacitor.Plugins`, JSON async message passing, `copy`-шаг.
- Документация Tauri v2 — процессная модель, `ipc://`, `invoke_key`, Commands/Events, permissions/scopes,
  channels для стримов.
- Документация Cordova — для сравнения и фиксации анти-паттернов.

> Замечание: точные имена пакетов и API Аврора (например, `ru.auroraos.webview`,
> `pkgconfig(aurorawebview)`, `libcef`) и доступность `CefMessageRouter`/`window.cefQuery`
> подлежат верификации на актуальной версии Aurora SDK на этапе MVP — см. [чек-листы](checklists.md).
